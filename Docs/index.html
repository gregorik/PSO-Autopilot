PSO Autopilot Manual

Welcome to PSO Autopilot. This plugin is the definitive, "AAA" solution to Unreal Engine 5's notorious shader compilation stutter.

Most "Shader Compilation" plugins on the marketplace attempt to solve stutter by brute-forcing the Engine—loading all of your project's assets into memory simultaneously. If you are building a massive 50GB open-world game, this primitive approach guarantees catastrophic RAM spikes, completely frozen loading screens, and eventually, Out-Of-Memory (OOM) crashes on consoles or lower-end PCs.

PSO Autopilot is different. It was engineered from the ground up to orchestrate Unreal's native hardware pipeline asynchronously, managing memory, threads, and driver caches with surgical precision.

The 3 Pillars of PSO Autopilot

Our architecture is built upon three Unique Selling Propositions (USPs) that completely destroy the competition.

USP 1: Memory-Safe Chunking (Materials & VFX)

Instead of loading thousands of assets simultaneously, PSO Autopilot utilizes Unreal's FStreamableManager to asynchronously stream your Materials and Niagara Systems into memory in strictly controlled batches (e.g., 100 assets at a time).

Once a batch is successfully compiled by the Engine's native FShaderPipelineCache, the plugin explicitly drops all asset references and requests a non-blocking Garbage Collection pass. The result? Your RAM footprint remains completely flat throughout the entire process, making this tool infinitely scalable for massive projects.

Batch 1 Loaded → PSOs Compiled → References Dropped → GC Sweeps → Batch 2 Loaded

USP 2: The Seamless Loading Screen

A frozen loading screen feels like a crash to a player. PSO Autopilot uses advanced C++ time-slicing within the Game Thread's Tick function. It processes Asset Registry queries and shader assignments for a strictly allocated timeframe per frame (e.g., 5.0 milliseconds).

If the time limit is reached, it yields execution back to the Engine. This guarantees that your loading screen videos, UI spinners, and progress bars remain buttery smooth at 60fps while intense compilation happens quietly in the background.

USP 3: Zero-Redundancy Boot (Smart Cache Hashing)

Players shouldn't have to wait for shaders they've already compiled. When a player boots your game for the second time, PSO Autopilot intelligently queries the local driver's cache state.

If it detects that the Pipeline State Objects (PSOs) for your project are already fully digested and stored on the user's hard drive from a previous play session, the plugin instantly skips the entire warmup sequence. A process that took two minutes on the first launch will take 0.05 seconds on all subsequent launches.

Quick Start Guide

Follow these steps to get PSO Autopilot running in your project in under 5 minutes:

  1. Ensure the PSOAutopilot folder is located in your project's Plugins directory.
  2. Launch your project in Unreal Engine 5.
  3. Go to Edit > Project Settings. Scroll down the left sidebar to the Plugins section and click on PSO Autopilot.
  4. Under Targeting, add the main directories containing your Materials and Niagara Systems (e.g., /Game/Characters, /Game/VFX).
  5. Open your primary Loading Screen Blueprint or GameInstance.
  6. Use the Get Game Instance Subsystem node and select PSOAutopilotSubsystem.
  7. Call the Start Warmup node.

Configuration Settings

PSO Autopilot exposes powerful variables to Project Settings so you can tailor the memory/performance balance precisely to your target hardware (PC vs. Console vs. Mobile).

Directories To Scan

An array of paths (e.g., /Game/Weapons). The plugin will recursively scan these folders for UMaterialInterface and UNiagaraSystem assets. Tip: Do not just put /Game/ if you have 100GB of audio and textures; isolate your heavy visual folders to speed up the initial Asset Registry query.

Batch Size

Default: 100. The maximum number of assets loaded into RAM simultaneously. Lower this to 50 if targeting mobile or low-RAM consoles to prevent memory spikes.

bGarbageCollectBetweenBatches

Default: True. Tells the Engine to asynchronously clean up unloaded memory after every batch. Only turn this off if you are managing GC manually in your own C++ code.

Max Processing Time Ms Per Frame

Default: 5.0. The time budget for the plugin's Game Thread operations. A 60fps frame takes 16.6ms. Leaving this at 5.0 gives the Engine 11.6ms to render your UI animations smoothly.

UI & Blueprint Integration

The PSOAutopilotSubsystem provides robust Blueprint delegates to drive your loading screen UI.

On Progress Updated

Fires continuously during the warmup phase.

  • Overall Progress (Float): A normalized 0.0 to 1.0 value representing the total completion percentage. Bind this directly to a UMG ProgressBar.
  • Current Status Message (String): Live telemetry of the internal state. It will cycle through messages like Scanning for Assets..., Loading Batch of 100..., Compiling PSOs... (42 remaining), and Clearing Memory....

On Warmup Complete

Fires exactly once when the entire process is finished (or if it is instantly skipped via USP 3). Bind this event to close your loading screen and load your primary gameplay level.

Setting up the Demo

PSO Autopilot includes built-in C++ architecture to create a "Torture Test" demo, allowing you to visually prove the USPs to your team or clients.

  1. Create a new level (e.g., Demo_PSOTorture).
  2. In your plugin's Content/Demo folder, create a Widget Blueprint named WBP_PSOAutopilotHUD and reparent it to the C++ class PSOAutopilotDemoWidget.
  3. Add the following elements to your widget, naming them exactly:
    • SpinnerImage (Image) - This will automatically rotate at 60fps to prove USP 2.
    • ProgressBar (ProgressBar) - Auto-binds to completion percentage.
    • StatusText (TextBlock) - Auto-binds to live telemetry.
    • TimerText (TextBlock) - Visualizes how fast USP 3 (Zero-Redundancy) skips loading.
  4. Drag a PSOAutopilotDemoManager actor into your level.
  5. In the Details panel of the Manager actor, set the HUDClass to your WBP_PSOAutopilotHUD.
  6. Point your Project Settings at a folder filled with massive, uncompiled assets, and press Play!

Troubleshooting

Q: The progress bar stays at 0% for a long time on boot.

A: Unreal Engine's Asset Registry initializes asynchronously in the background when the game boots. PSO Autopilot intelligently waits for this background thread to finish before querying assets to ensure none are missed. If your project is massive, this initial wait is normal.

Q: I see "Compiling PSOs..." in the log, but no shaders are actually building.

A: Ensure your project is configured to actually generate shader caches. In your DefaultProject.ini, verify that bShareMaterialShaderCode is enabled, and test in a Packaged Build. The Editor environment handles PSOs differently than a deployed executable.
